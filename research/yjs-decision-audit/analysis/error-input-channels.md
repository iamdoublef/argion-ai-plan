# 错误信息入口：5个够了

> 日期：2026-02-25
> 定位：从18个入口砍到5个，每个都说清楚为什么选它
> 约束：内部无开发人员，仅技术协助方一人可执行技术工作
> 原则：管理者亲口说的痛点 > 推演出来的需求；数据现成的 > 需要建制度的

---

## 一、选择标准

不是拍脑袋选的。每个入口必须同时满足以下5条：

1. **直接对应管理者录音中的真实痛点**——不是推演出来的，是管理者亲口说的
2. **数据已经存在或马上能拿到**——不需要建制度、改文化、等人配合
3. **一个人能干**——大boss手上只有一个技术协助方
4. **事前拦截 + 事后评分形成闭环**——光拦不考核没人有动力，光考核没数据是空壳
5. **投入产出比可量化**——能算清楚省多少钱或避免多少损失

---

## 二、Top 5 入口

### 第1名：B1 产品说明书AI核查

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前：发布前AI扫描，错误在出厂前拦截 |
| 数据可得性 | 高。说明书是成品文档，通常有电子版 [待确认格式] |
| 接入难度（一人） | 极低。LLM API一个调用即可完成核查 |
| 可预警场景 | 单位制式错误、翻译质量、合规标识缺失、与历史错误模式匹配 |

**为什么排第一：**

管理者录音里的案例B就是这个——说明书单位错误，连续两年重复犯，工程师→经理→副总三层审核全部失守。

这是18个入口里唯一一个**技术上半天就能搞定**的：一个LLM API调用，输入说明书PDF，输出错误清单。纯事前拦截——说明书出厂前过一遍AI，错了就打回去，不需要任何人配合。修复成本极低：改一份说明书可能就10分钟，但出错后的客户投诉、品牌损伤是不可逆的。不需要确认任何系统——只要能拿到说明书电子版就行。

**管理之神点评：** 这是送分题。三层审核全失守，说明人不靠谱，那就让机器来。一个API调用，半天上线，零依赖。18个入口里找不到比这更划算的了。

---

### 第2名：A2 电商后台数据（Amazon Seller Central）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：库存水位、退货率、差评趋势可日频监控 |
| 数据可得性 | 高。Amazon提供Seller Central报表下载，SP-API可编程访问 |
| 接入难度（一人） | 低-中。手动下载报表→低；API自动化→中（需开发者账号） |
| 可预警场景 | FBA库存超龄预警、退货率突增、Listing被抑制、广告ACOS异常 |

**为什么排第二：**

管理者录音里的案例A就是这个——库存决策失误，同一个人在美国犯了一次，若干年后在欧洲又犯一次。

Amazon Seller Central的数据是**现成的**，不需要任何开发，登录后台就能下载FBA库存报告、长期仓储费报告。库存超龄180天的数据一拉出来，哪些SKU该销毁、哪些该促销，一目了然。这是**事中预警**：库存在变呆滞的过程中就能发现，不用等到年底盘点才知道亏了多少。亚俊氏的自有品牌Wevac/Vesta主要在Amazon卖，这个入口直接覆盖自有品牌业务线。

**管理之神点评：** 库存是制造企业的命。我见过太多工厂，产品做得好，最后死在库存上。呆滞库存不是"放着也没事"，它占资金、占仓位、还有仓储费在烧。管理者说的"从未考虑过销毁"，说明这个高管的库存管理意识有盲区，数据能帮他看到盲区。

---

### 第3名：A1 ERP系统数据（库存/采购/生产）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前+事中：库存异常、采购偏差、排产冲突可实时监控 |
| 数据可得性 | 取决于ERP品牌和版本 [待确认]。金蝶/用友通常支持数据导出（Excel/API） |
| 接入难度（一人） | 中。如有API→中低；如只能导出Excel→低（定期导出+脚本分析） |
| 可预警场景 | 呆滞库存超阈值告警、采购量vs销售预测偏差、原材料价格异常波动 |

**为什么排第三：**

ERP是制造企业的"中枢神经"，库存周转天数、采购偏差、排产冲突全在里面。和A2（电商数据）形成闭环：A2看的是"卖端"库存，A1看的是"产端"库存，两头一对就知道供需是否匹配。即使ERP只能导出Excel（大概率），定期导出+Python脚本分析也够用。覆盖的决策场景最广：采购多少料、排多少产、备多少货，全是高管日常决策。

**为什么不排更前：** 因为ERP品牌和版本还不确定 [待确认]。如果是金蝶/用友，数据导出很方便；如果是老旧系统，可能要费点劲。所以排在B1和A2后面——那两个不依赖任何确认就能启动。

**管理之神点评：** 600人的厂，ERP里的数据量不会太大，但信息密度很高。我在工厂管了10年，最怕的就是"采购说备了料，生产说没收到，仓库说早发了"——这种扯皮ERP数据一拉就清楚。

---

### 第4名：B5 客诉记录

| 维度 | 评估 |
|------|------|
| 预警时机 | 事后（已发生），但可预警趋势恶化 |
| 数据可得性 | 中。Amazon差评/退货数据现成；ODM客户投诉记录企业内部应有 [待确认] |
| 接入难度（一人） | 低-中。如有结构化记录→低；如分散在邮件中→中 |
| 可预警场景 | 同类客诉重复出现、客诉率突增、大客户投诉升级、退货原因追溯 |

**为什么排第四：**

客诉是决策错误的**直接后果**——说明书错了客户会投诉，库存策略错了退货率会飙升。Amazon差评和退货原因数据是现成的（和A2联动），ODM客户的投诉记录企业内部也应该有。客诉数据的价值在于**追溯**：同类客诉反复出现，说明根因没解决，可以直接关联到责任人。这是把"事后数据"变成"事前预警"的关键入口：上个月退货率突增10%，这个月同款产品还在出货，该不该拦？

**为什么不排更前：** 客诉本质是事后数据，错误已经发生了。但它的价值在于"防止同样的错误再犯"——和管理者的核心诉求（重复错误识别）完全吻合。

**管理之神点评：** 客诉是最诚实的反馈。客户不会跟你客气，哪里不好直接说。很多工厂把客诉当"灭火"——投诉来了赶紧处理，处理完就忘了。真正该做的是把客诉当"体检报告"——同一个毛病反复出现，说明根上有问题。

### 第5名：错误台账 + 事后评分（考核联动）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事后汇聚：前4个入口的错误数据都往这里灌 |
| 数据可得性 | 依赖前4个入口提供数据 |
| 接入难度（一人） | 低。飞书多维表格即可实现 |
| 可预警场景 | 高管决策质量评分、重复错误标记、压力传导（副总→经理→工程师） |

**为什么排第五（但它是整个系统的发动机）：**

管理者原话："最无情的考核就是AI"——去除人情因素，用数据说话。管理者要的压力传导：副总被考核 → 副总主动盯经理 → 经理主动盯工程师。**没有事后评分，前4个入口采集到的数据就是死数据**——副总知道"错了也没事"，他根本不会去看预警报告。

这不是一个独立的"信息入口"，它是**汇聚层**：前4个入口的错误数据都往这里灌，输出每个高管的"决策质量评分卡"。技术上用飞书多维表格就能实现：错误记录→自动统计→季度报告→年度考核。

它解决的核心问题：
- 管理者记不住谁犯过什么错 → 台账就是记忆库
- 同样错误反复犯 → 重复标记+自动关联历史记录
- 副总护短 → 副总自己也被评分，有压力传导
- 觉得被针对 → 制度公开透明，数据说话，不是人盯人

**为什么排最后而不是第一：** 因为台账需要数据喂养。没有前4个入口提供的错误数据，台账就是空壳。先把数据源接通（B1/A2/A1/B5），台账才有东西可以统计。但一旦数据开始流入，评分系统就是让整个机器转起来的钥匙。

**管理之神点评：** 我管了30年工厂，见过无数"好系统没人用"的案例。为什么没人用？因为用不用都一样，没有后果。你把考核挂上去，副总每个季度看到自己的评分卡，上面写着"本季度严重错误1次（库存决策），重复错误1次（与2025年案例相同），建议绩效系数0.82"——你看他下次开会讨论库存的时候认不认真。

## 三、砍掉的13个入口

| 入口 | 砍掉理由 |
|------|---------|
| A3 OA审批流 | 依赖OA类型 [待确认]，且审批数据只能间接推断判断质量 |
| A4 邮件/即时通讯 | 噪音太大，信息分散，一个人处理不过来 |
| A5 财务系统 | 事后数据为主，且财务数据敏感需授权，优先级不够 |
| B2 合同/报价单 | 重要但频率低，ODM合同一年就那么几份，不是日常预警场景 |
| B3 质检报告 | 质检数据记录方式未知，可能是纸质，数字化成本高 |
| B4 工程图纸/BOM | 需要专业知识判断，AI目前做不好，且频率低 |
| C1 会议纪要 | 被D1覆盖，录音比纪要更完整更客观 |
| C2 产线巡检 | 与高管决策错误关联度低，是车间管理层面的事 |
| C3 非正式客户反馈 | 数据获取极难，存在销售人员脑中，建机制成本高 |
| C4 员工反馈/吹哨 | 需要文化变革支撑，短期内做不到 |
| D1 决策会议录音 | 价值高但需管理者配合+文化适应，列为Top 5跑顺后的第一个扩展项 |
| D2 审批判断 | 数据可得性极低，只能间接推断 |
| D3 供应商选择 | 频率低，不是日常预警场景 |
| D4 市场判断 | 太主观，AI难以判断"市场判断是否正确" |

## 四、实施路径（3个阶段）

一个技术协助方，3个月见效。

### 第一阶段：快速验证（第1-2周）

目标：让管理者看到"AI能帮我发现问题"。

```
动作1：说明书AI核查（B1）
  - 收集最近3份出口美国的产品说明书
  - 用LLM API检查：单位制式、翻译质量、合规标识
  - 输出核查报告给管理者看效果
  - 工具：ChatGPT/Claude API，成本≈$0.1/份

动作2：Amazon数据快照（A2）
  - 下载FBA库存报告、长期仓储费报告
  - 标记呆滞库存（>180天）和超龄库存
  - 输出一页纸预警摘要
  - 工具：Amazon Seller Central手动下载 + Excel/Python
```

### 第二阶段：数据汇聚（第3-6周）

目标：建立错误信息的"单一入口"，数据开始流动。

```
动作3：错误台账上线
  - 飞书/钉钉多维表格，字段见mvp-proposal.md
  - 管理者发现错误→2分钟录入→自动统计
  - B1核查结果自动灌入台账

动作4：ERP数据接入（A1）[取决于确认结果]
  - 确认ERP品牌和导出能力
  - 设置周频导出：库存报表、采购报表、生产报表
  - Python脚本自动分析异常值

动作5：客诉数据整理（B5）
  - 收集现有客诉记录（不管什么格式）
  - Amazon差评+退货原因数据与A2联动
  - 按产品/问题类型/责任人分类，灌入台账
```

### 第三阶段：闭环运转（第7-12周）

目标：从"人发现问题"升级到"AI发现问题+考核跟上"。

```
动作6：说明书核查流程化
  - 所有出厂说明书必须经过AI核查
  - 核查结果自动录入错误台账
  - 建立历史错误模式库

动作7：库存预警自动化
  - Amazon + ERP数据定期自动分析
  - 设置阈值告警（呆滞库存、库存周转异常）
  - 预警结果推送给管理者

动作8：事后评分上线
  - 台账数据自动生成季度评分卡
  - 每个高管的"决策质量评分"：错误次数、严重程度、重复率
  - 评分结果与绩效考核挂钩
```

## 五、需要企业确认的信息

| # | 待确认项 | 影响的入口 | 影响程度 |
|---|---------|-----------|---------|

## 六、下一步扩展（Top 5跑顺之后）

Top 5跑顺后，第一个扩展项是 **D1 决策会议录音+AI分析**。价值最高（直接捕获决策过程），但需要管理者配合录音+文化适应，所以不放在第一批。等台账运行1-2个月、管理者看到数据的价值后，再推这个阻力会小很多。
