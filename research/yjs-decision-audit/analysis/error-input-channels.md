# 错误信息入口全景分析

> 日期：2026-02-25
> 定位：系统梳理亚俊氏所有可能的错误/决策信息入口，评估优先级
> 约束：内部无开发人员，仅技术协助方一人可执行技术工作
> 重点：事前/事中预警，非事后追责

---

## 二、逐入口分析

### A类：数字系统入口

#### A1. ERP系统（进销存/生产）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前+事中：库存异常、采购偏差、排产冲突可实时监控 |
| 数据可得性 | 取决于ERP品牌和版本 [待确认]。金蝶/用友通常支持数据导出（Excel/API） |
| 接入难度（一人） | 中。如有API→中低；如只能导出Excel→低（定期导出+脚本分析） |
| 预警价值 | ⭐⭐⭐⭐⭐ 库存决策失误是管理者提到的核心痛点，ERP是第一手数据源 |
| 亚俊氏案例 | 库存积压未及时销毁（案例A）→ ERP中的库存周转天数、呆滞库存报表可直接预警 |
| 可预警场景 | 呆滞库存超阈值告警、采购量vs销售预测偏差、原材料价格异常波动 |

#### A2. 电商后台（Amazon Seller Central等）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：库存水位、退货率、差评趋势可日频监控 |
| 数据可得性 | 高。Amazon提供Seller Central报表下载，SP-API可编程访问 |
| 接入难度（一人） | 低-中。手动下载报表→低；API自动化→中（需开发者账号） |
| 预警价值 | ⭐⭐⭐⭐⭐ 直接关联库存决策和市场判断，是案例A的核心数据源 |
| 亚俊氏案例 | 亚马逊库存策略失误 → FBA库存报表、长期仓储费报表可直接预警 |
| 可预警场景 | FBA库存超龄预警、退货率突增、Listing被抑制、广告ACOS异常 |

#### A3. OA系统（审批流）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：审批超时、流程跳过、异常审批可实时捕获 |
| 数据可得性 | 取决于OA类型 [待确认]。钉钉/企业微信/飞书均有开放API |
| 接入难度（一人） | 中。如用钉钉/飞书→API文档完善，可自行对接；如用传统OA→难 |
| 预警价值 | ⭐⭐⭐ 审批流是"审核失守"的直接证据（案例B中经理审核形同虚设） |
| 亚俊氏案例 | 说明书审核流程多层失守 → OA中审批时长、是否实际查看附件可分析 |
| 可预警场景 | 审批秒过（未实际审核）、流程跳级、同类审批反复驳回 |

#### A4. 邮件/即时通讯

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：客户投诉邮件、供应商异常通知可实时捕获 |
| 数据可得性 | 中。企业邮箱可设置转发/归档；即时通讯（微信）数据获取困难 |
| 接入难度（一人） | 中-高。邮件转发规则→低；系统化分析→需NLP处理 |
| 预警价值 | ⭐⭐ 信息分散、噪音大，但包含客户直接反馈 |
| 亚俊氏案例 | 国际客户对说明书单位问题的投诉可能最先出现在邮件中 |
| 可预警场景 | 客户投诉关键词监控、供应商交期延迟通知、内部升级邮件 |

#### A5. 财务系统

| 维度 | 评估 |
|------|------|
| 预警时机 | 事后为主，但异常支出可事中预警 |
| 数据可得性 | 中。财务数据敏感，通常需要财务总监授权 [待确认] |
| 接入难度（一人） | 中。通常只能获取汇总报表，明细数据需授权 |
| 预警价值 | ⭐⭐⭐ 财务异常是决策错误的最终体现（如库存减值损失） |
| 亚俊氏案例 | 呆滞库存的仓储费和减值损失，在财务报表中有直接体现 |
| 可预警场景 | 单项费用超预算、毛利率异常下降、应收账款超期 |

### B类：文档流程入口

#### B1. 产品说明书/包装文案

| 维度 | 评估 |
|------|------|
| 预警时机 | ⭐事前：发布前AI扫描，错误在出厂前拦截 |
| 数据可得性 | 高。说明书是成品文档，通常有电子版 [待确认格式] |
| 接入难度（一人） | ⭐极低。LLM API一个调用即可完成核查 |
| 预警价值 | ⭐⭐⭐⭐ 管理者明确提到的痛点（案例B），且修复成本极低 |
| 亚俊氏案例 | 美国市场说明书公制/英制单位错误，重复发生 |
| 可预警场景 | 单位制式错误、翻译质量、合规标识缺失、与历史错误模式匹配 |

#### B2. 合同/报价单

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前：签约前AI核查条款异常 |
| 数据可得性 | 中。合同通常有电子版，但可能分散存储 [待确认] |
| 接入难度（一人） | 低-中。PDF/Word文档→LLM可直接处理 |
| 预警价值 | ⭐⭐⭐ 合同条款错误的损失可能很大（价格、交期、质量条款） |
| 亚俊氏案例 | ODM客户合同中的价格、MOQ、交期条款是否合理 |
| 可预警场景 | 价格低于成本线、付款条件异常、质量条款过严、交期不合理 |

#### B3. 质检报告

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：质检数据异常趋势可实时预警 |
| 数据可得性 | 取决于质检记录方式 [待确认]。可能是纸质/Excel/MES系统 |
| 接入难度（一人） | 低（如Excel）到高（如纸质需OCR） |
| 预警价值 | ⭐⭐⭐ 质量问题是制造企业的核心风险 |
| 亚俊氏案例 | 注塑件不良率、封口质量波动 |
| 可预警场景 | 不良率突增、特定产线/班次异常、同一缺陷重复出现 |

#### B4. 工程图纸/BOM表

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前：新品投产前核查 |
| 数据可得性 | 中。CAD文件格式多样，BOM可能在ERP或Excel中 [待确认] |
| 接入难度（一人） | 中-高。BOM比对→中；图纸审查→高（需专业工具） |
| 预警价值 | ⭐⭐ 重要但频率低，且需要专业知识判断 |
| 亚俊氏案例 | ODM客户定制产品的BOM变更是否正确执行 |
| 可预警场景 | BOM版本不一致、物料替代未经审批、成本核算偏差 |

#### B5. 客诉记录

| 维度 | 评估 |
|------|------|
| 预警时机 | 事后（已发生），但可预警趋势恶化 |
| 数据可得性 | 中。可能在CRM、Excel或邮件中 [待确认] |
| 接入难度（一人） | 低-中。如有结构化记录→低；如分散在邮件中→中 |
| 预警价值 | ⭐⭐⭐⭐ 客诉是决策错误的直接后果，且关联客户流失风险 |
| 亚俊氏案例 | Amazon差评和退货原因分析，可追溯到产品/说明书决策 |
| 可预警场景 | 同类客诉重复出现、客诉率突增、大客户投诉升级 |

### C类：线下人工入口

#### C1. 会议纪要

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：决策会议中的分歧和风险信号 |
| 数据可得性 | 低-中。取决于是否有记录习惯 [待确认] |
| 接入难度（一人） | 中。如有录音→AI转写+分析；如无记录→需建立制度 |
| 预警价值 | ⭐⭐⭐⭐ 决策过程的直接记录，可追溯"谁说了什么" |
| 亚俊氏案例 | 管理者提到的"集体决策记录"需求（M1），库存策略讨论会 |
| 可预警场景 | 决策未经充分讨论、反对意见被忽略、执行偏离决策 |

#### C2. 产线巡检/现场记录

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：现场异常实时发现 |
| 数据可得性 | 低。通常是口头或纸质 [待确认] |
| 接入难度（一人） | 高。需要建立数字化巡检流程（如用钉钉表单） |
| 预警价值 | ⭐⭐ 对制造企业重要，但与高管决策错误的关联度较低 |
| 亚俊氏案例 | 广州/佛山产线的设备状态、5S执行情况 |
| 可预警场景 | 设备异常、安全隐患、工艺偏差 |

#### C3. 客户反馈（非正式）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：客户口头反馈往往是最早的预警信号 |
| 数据可得性 | 极低。存在销售人员脑中，未被记录 |
| 接入难度（一人） | 高。需要建立反馈录入机制 |
| 预警价值 | ⭐⭐⭐ 客户非正式反馈常常预示正式投诉 |
| 亚俊氏案例 | ODM客户对交期/质量的口头不满，可能预示订单流失 |
| 可预警场景 | 客户满意度下降趋势、竞品替代风险信号 |

#### C4. 员工反馈/吹哨

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前/事中：一线员工最先发现问题 |
| 数据可得性 | 极低。员工不敢说、不愿说、没渠道说 |
| 接入难度（一人） | 中。可建立匿名反馈渠道（如匿名表单） |
| 预警价值 | ⭐⭐⭐ 可发现管理层看不到的问题，但需要文化支撑 |
| 亚俊氏案例 | 管理者提到的"护短文化"——员工可能知道问题但不敢上报 |
| 可预警场景 | 流程违规、质量隐瞒、管理层决策盲区 |

### D类：无形决策入口

#### D1. 决策会议（战略/运营）

| 维度 | 评估 |
|------|------|
| 预警时机 | ⭐事前：决策做出前的最后防线 |
| 数据可得性 | 低。需要主动记录 |
| 接入难度（一人） | 中。录音+AI转写→可行；结构化分析→需定义框架 |
| 预警价值 | ⭐⭐⭐⭐⭐ 直接捕获决策过程，是"重复错误识别"的核心数据源 |
| 亚俊氏案例 | 库存策略决策会、新品立项会、市场进入决策 |
| 可预警场景 | 决策者未考虑历史教训、关键风险未被讨论、决策依据不充分 |

#### D2. 审批判断（日常）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事中：审批时的判断质量 |
| 数据可得性 | 极低。审批人脑中的判断过程无法直接获取 |
| 接入难度（一人） | 高。只能通过审批结果间接推断（如秒批=未审核） |
| 预警价值 | ⭐⭐⭐ 案例B中"审核形同虚设"的核心问题 |
| 亚俊氏案例 | 说明书审核流程中经理和副总的审批质量 |
| 可预警场景 | 审批时间异常短、审批后仍出错（说明未实际审核） |

#### D3. 供应商选择/评估

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前：选择供应商前的评估质量 |
| 数据可得性 | 低-中。如有供应商评估表→中；如凭经验→低 [待确认] |
| 接入难度（一人） | 中。需要建立评估模板和历史数据库 |
| 预警价值 | ⭐⭐ 重要但频率低 |
| 亚俊氏案例 | 原材料供应商选择、物流服务商评估 |
| 可预警场景 | 供应商集中度过高、价格偏离市场、质量记录不佳 |

#### D4. 市场判断（定价/渠道/新品）

| 维度 | 评估 |
|------|------|
| 预警时机 | 事前：市场决策前的数据支撑 |
| 数据可得性 | 中。电商数据可获取，但市场判断本身是主观的 |
| 接入难度（一人） | 中。可用电商数据+竞品监控辅助判断 |
| 预警价值 | ⭐⭐⭐ 市场判断错误的损失可能很大（如欧洲市场库存案例） |
| 亚俊氏案例 | 欧洲市场库存策略重复失误、新品定价决策 |
| 可预警场景 | 定价偏离竞品区间、库存备货与销售趋势背离、新市场进入时机 |

## 三、优先级矩阵

按"预警价值"和"一个人的接入难度"两个维度，将18个入口分为四象限：

```
                        预警价值 高
                            │
        ② 值得投入            │         ① 立即做
        (高价值 + 高难度)      │         (高价值 + 低难度)
                            │
   D1 决策会议录音+AI分析     │    B1 产品说明书AI核查 ★
   C1 会议纪要数字化          │    A2 电商后台数据监控 ★
   A1 ERP数据分析（如需API）   │    A1 ERP报表导出分析 ★
   C4 员工匿名反馈渠道        │    B5 客诉记录汇总分析
                            │    B2 合同/报价单AI核查
 ──────────────────────────┼──────────────────────────
                            │
        ④ 暂不做              │         ③ 有空再做
        (低价值 + 高难度)      │         (低价值 + 低难度)
                            │
   C2 产线巡检数字化          │    A5 财务报表异常监控
   B4 工程图纸/BOM核查        │    D3 供应商评估模板
   D2 审批判断质量分析         │    A4 邮件关键词监控
   C3 非正式客户反馈          │
                            │
                        预警价值 低
```

### 第一象限详解（立即做）

| 排序 | 入口 | 理由 | 一个人能做吗 | 预计启动时间 |
|------|------|------|-------------|------------|
| 1 | B1 说明书AI核查 | 管理者明确痛点+技术最简单（一个LLM调用） | ✅ 半天搭建 | 本周 |
| 2 | A2 电商数据监控 | 库存决策核心数据源+Amazon报表免费下载 | ✅ 1-2天写脚本 | 本周 |
| 3 | A1 ERP报表分析 | 库存/采购核心数据+导出Excel即可分析 | ✅ 取决于ERP [待确认] | 确认ERP后 |
| 4 | B5 客诉汇总 | 决策错误的直接后果+数据可能已有 | ✅ 1天整理 | 本月 |
| 5 | B2 合同AI核查 | 高损失风险+LLM可直接处理 | ✅ 1天搭建 | 本月 |

### 第二象限详解（值得投入）

| 排序 | 入口 | 理由 | 难点 | 建议时机 |
|------|------|------|------|---------|
| 1 | D1 决策会议录音 | 最高预警价值，直接捕获决策过程 | 需要管理者配合录音+AI转写成本 | 台账运行1月后 |
| 2 | C1 会议纪要 | 决策追溯的关键证据 | 需要建立记录制度 | 与D1同步 |
| 3 | A1 ERP API对接 | 实时监控vs手动导出 | 需要ERP厂商配合 [待确认] | 手动验证有效后 |
| 4 | C4 员工反馈 | 发现管理盲区 | 需要文化支撑，匿名机制 | 制度建立后 |

## 四、推荐实施路径

考虑约束：一个技术协助方、无内部开发人员、管理者需要快速看到效果。

### 第一阶段：快速验证（第1-2周）

目标：让管理者看到"AI能帮我发现问题"。

```
动作1：说明书AI核查（B1）
  - 收集最近3份出口美国的产品说明书
  - 用LLM API检查：单位制式、翻译质量、合规标识
  - 输出核查报告给管理者看效果
  - 工具：ChatGPT/Claude API，成本≈$0.1/份

动作2：Amazon数据快照（A2）
  - 下载FBA库存报告、长期仓储费报告
  - 标记呆滞库存（>180天）和超龄库存
  - 输出一页纸预警摘要
  - 工具：Amazon Seller Central手动下载 + Excel/Python
```

### 第二阶段：数据汇聚（第3-4周）

目标：建立错误信息的"单一入口"。

```
动作3：错误台账上线（与MVP方案联动）
  - 飞书/钉钉多维表格，字段见mvp-proposal.md
  - 管理者发现错误→2分钟录入→自动统计

动作4：ERP数据接入（A1）[取决于确认结果]
  - 确认ERP品牌和导出能力
  - 设置周频导出：库存报表、采购报表、生产报表
  - Python脚本自动分析异常值

动作5：客诉数据整理（B5）
  - 收集现有客诉记录（不管什么格式）
  - 统一录入结构化表格
  - 按产品/问题类型/责任人分类
```

### 第三阶段：AI预警（第2-3月）

目标：从"人发现问题"升级到"AI发现问题"。

```
动作6：说明书核查流程化
  - 所有出厂说明书必须经过AI核查
  - 核查结果自动录入错误台账
  - 建立历史错误模式库

动作7：库存预警自动化
  - Amazon + ERP数据定期自动分析
  - 设置阈值告警（呆滞库存、库存周转异常）
  - 预警结果推送给管理者

动作8：合同/报价单核查（B2）
  - 重要合同签约前AI核查
  - 检查价格、条款、交期合理性
```

### 第四阶段：深度预警（第4-6月）

目标：捕获"无形决策"中的错误信号。

```
动作9：决策会议录音+AI分析（D1）
  - 重要决策会议录音（需管理者授权）
  - AI转写+提取决策要点
  - 与历史错误库比对：是否在重复过去的错误模式

动作10：审批质量分析（D2，间接方式）
  - 从OA系统导出审批记录 [待确认OA类型]
  - 分析审批时长分布
  - 标记"秒批"（<1分钟）的高风险审批
```

一个技术协助方，6个月内可实现的预警覆盖：

| 阶段 | 覆盖入口数 | 累计预警能力 | 管理者感知 |
|------|-----------|------------|-----------|
| 第1阶段（2周） | 2个（B1+A2） | 说明书错误拦截 + 库存异常预警 | "AI确实能发现问题" |
| 第2阶段（1月） | 5个（+A1+B5+台账） | 错误有记录、有统计、有追溯 | "终于有人帮我记住了" |
| 第3阶段（3月） | 7个（+B2+自动化） | 关键文档出厂前必过AI、库存自动预警 | "问题在发生前就被拦住了" |

最终目标：18个入口中覆盖9个高价值入口，剩余9个在团队扩充后逐步接入。
