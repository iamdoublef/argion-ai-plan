# 局部先行方案 — 核心高管决策质量追踪（MVP）

> 日期：2026-02-25
> 定位：不上大系统，先解决3-5个核心高管的重复犯错问题
> 原则：最小工具 + 最快见效 + 管理者能直接用

---

## 一、为什么局部先行

管理者的痛点不是60个人都有问题，而是：
- **2-3个副总** 反复犯同样的决策错误（库存、说明书等）
- **几个部门经理** 审核形同虚设
- 管理者没有一个"记忆系统"帮他记住谁犯过什么错

核心需求其实是：**给管理者一个"高管错误记忆库"，让数据替他说话。**

## 二、MVP方案：高管决策错误台账

### 2.1 工具选择

不需要开发系统，用现有工具即可启动：

| 方案 | 工具 | 成本 | 上手时间 |
|------|------|------|---------|
| A（推荐） | 飞书/钉钉多维表格 | 免费 | 1天 |
| B | Excel + 共享网盘 | 免费 | 半天 |
| C | 简单Web应用 | 开发成本 | 2-4周 |

推荐方案A：飞书多维表格，支持表单录入、自动统计、权限控制、移动端。

### 2.2 台账字段设计

每发现一个错误，录入一条记录：

| 字段 | 类型 | 说明 |
|------|------|------|
| 日期 | 日期 | 发现日期 |
| 错误描述 | 文本 | 一句话描述发生了什么 |
| 业务领域 | 单选 | 库存/产品/市场/供应链/文档/人事 |
| 错误类型 | 单选 | 遗漏（该想没想到）/ 执行（做错了）/ 流程（审核失守） |
| 严重度 | 单选 | 严重 / 次要 / 一般 |
| 直接责任人 | 人员 | 谁犯的错 |
| 审核责任人 | 人员 | 谁该查出来但没查出来 |
| 分管领导 | 人员 | 哪个副总分管 |
| 是否重复 | 是/否 | 这个人以前犯过类似错误吗 |
| 关联历史记录 | 关联 | 如果是重复，关联到上次的记录 |
| 处理结果 | 文本 | 怎么处理的 |
| 备注 | 文本 | 补充信息 |

### 2.3 覆盖人员（第一批）

只追踪核心管理层，建议3-5人：

| 角色 | 追踪重点 |
|------|---------|
| 副总A（分管运营/市场） | 库存决策、市场策略决策 |
| 副总B（分管产品/研发） | 产品设计决策、文档质量 |
| 经理X（运营部） | 库存执行、供应链决策 |
| 经理Y（产品部） | 说明书审核、产品规格 |
| （可选）经理Z | 根据实际情况增加 |

### 2.4 录入流程

```
发现错误（管理者/其他管理层/AI核查）
    │
    ▼
填写台账表单（2分钟）
    │
    ▼
系统自动检查：该责任人是否有历史类似记录
    │
    ├── 有 → 标记"重复错误"，关联历史记录
    │
    └── 无 → 正常记录
    │
    ▼
季度/年度汇总 → 考核依据
```

## 三、AI增强（第二步，台账跑顺后加）

台账运行1-2个月后，积累了数据，再加AI能力：

### 3.1 说明书自动核查（技术协助方评估可快速实现）

```
输入：产品说明书（PDF/Word）
AI检查项：
  ✓ 单位制式（美国市场→英制优先）
  ✓ 目标市场语言规范
  ✓ 与历史错误模式匹配（如上次漏了英制，这次是否又漏了）
输出：核查报告 + 问题清单
```

这个确实简单，一个LLM API调用就能做。

### 3.2 重复错误自动识别

当录入新错误时，AI自动：
- 搜索该责任人的历史记录
- 比对业务领域 + 错误类型
- 如果匹配 → 自动标红 + 推送提醒给管理者

### 3.3 季度考核报告自动生成

AI读取台账数据，自动生成：
```
[副总A] 2026年Q1 决策质量报告
━━━━━━━━━━━━━━━━━━━━━━━━━━
错误统计：严重1次 / 次要2次 / 一般1次
重复错误：1次（库存决策，与2025年案例相同）
责任分布：
  - 作为直接决策者：2次
  - 作为分管领导（下属犯错）：2次
建议绩效系数：0.82
━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 四、考核挂钩（简化版）

不需要复杂公式，先用简单规则：

| 年度情况 | 考核影响 |
|---------|---------|
| 0次严重 + ≤2次次要 | 正常发放奖金 |
| 1次严重 或 3次以上次要 | 奖金扣10-20% |
| 任何重复严重错误 | 奖金扣30%+ 谈话警告 |
| 连续两年有重复严重错误 | 降薪或调岗评估 |

关键：规则提前公布，所有人签字确认，不搞秋后算账。

## 五、实施时间线

```
第1周：  设计台账表格 + 定义错误分类标准
         ↓
第2周：  与核心高管沟通制度（透明公开）
         所有人签字确认考核规则
         ↓
第3周起：开始录入，试运行
         ↓
第2-3月：积累数据，验证分类标准是否合理
         ↓
第3月：  加入AI说明书核查（快速开发）
         ↓
第6月：  首次半年度考核报告
         根据数据决定是否扩大范围
```

## 六、这个方案解决了什么

| 管理痛点 | 怎么解决 |
|---------|---------|
| 记不住谁犯过什么错 | 台账就是记忆库 |
| 同样错误反复犯 | 重复标记+自动关联 |
| 管不过来 | 数据自动统计，不用人盯 |
| 觉得被针对 | 制度公开透明，数据说话 |
| 副总护短 | 副总自己也被追踪，有压力传导 |
| 说明书反复出错 | AI自动核查（第二步） |

## 七、不做什么（明确边界）

- 不做集体决策记录系统（太重，后面再说）
- 不做ERP/OA对接（先独立运行）
- 不做全员覆盖（先3-5个核心人员）
- 不做复杂的RPN评分（先用简单三级分类）
