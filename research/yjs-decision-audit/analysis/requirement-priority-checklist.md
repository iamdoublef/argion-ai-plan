# 管理需求确认清单：一条主线串到底

> 日期：2026-02-25
> 来源：管理者会议录音 → d0-research-brief.md
> 用途：大boss确认后，按此主线推进实施

---

## 管理者的终极目标

**不用亲自盯，副总自己有动力管好决策质量。**

怎么实现？靠考核。考核需要数据，数据需要入口。

整个系统就是一条链：

```
数据入口（采集错误证据）
    ↓
错误台账（记录 + 分级 + 追溯责任链）
    ↓
评分卡（自动统计每个高管的决策质量）
    ↓
考核挂钩（评分影响绩效/奖金）—— 发动机
    ↓
压力传导（副总→经理→工程师自驱动）—— 终局
```

下面按这条链，逐个环节说清楚：做什么、现在什么状态、需要确认什么。

---

## 环节1：数据入口（采集错误证据）

> 对应需求：F6 文档自动核查、F1 决策错误记录（数据来源部分）
>
> 这是整条链的起点。入口不通，后面全白搭。

### 入口1：产品说明书（B1）

- **管理者场景**：案例B——说明书单位错误连续两年，三层审核全失守
- **怎么采集**：说明书电子版 → LLM API核查 → 输出错误清单
- **现在能做吗**：技术上半天搞定，零依赖
- **卡在哪**：需要拿到说明书电子版
- **要确认的**：说明书电子版在哪？什么格式？谁手上有？
- [ ] **确认：说明书电子版可以拿到**

### 入口2：Amazon电商数据（A2）

- **管理者场景**：案例A——库存决策失误，同一人美国犯一次欧洲又犯一次
- **怎么采集**：登录Seller Central下载FBA库存报告、长期仓储费报告
- **现在能做吗**：数据现成，登录就能下载
- **卡在哪**：需要账号权限
- **要确认的**：账号权限谁有？有几个站点（美国/欧洲）？
- [ ] **确认：Amazon账号权限可以拿到**

### 入口3：ERP系统数据（A1）

- **管理者场景**：采购偏差、排产冲突、库存异常，制造企业的中枢神经
- **怎么采集**：导出Excel或API对接 → Python脚本分析异常值
- **现在能做吗**：取决于ERP品牌和导出能力
- **卡在哪**：ERP品牌和版本未知
- **要确认的**：用什么ERP？能导出数据吗？导出权限谁有？
- [ ] **确认：ERP品牌和导出方式**

### 入口4：客诉记录（B5）

- **管理者场景**：客诉是决策错误的直接后果，同类客诉反复出现说明根因没解决
- **怎么采集**：Amazon差评+退货数据（和入口2联动）+ ODM客户投诉记录
- **现在能做吗**：Amazon部分现成，ODM部分取决于记录方式
- **卡在哪**：不知道客诉记录在哪、什么格式
- **要确认的**：客诉记录在哪？系统/Excel/邮件/纸质？ODM客户投诉谁管？
- [ ] **确认：客诉数据的存放位置和格式**

### 入口5：错误台账（新建）

- **说明**：这是我们自己建的汇聚层，前4个入口的数据都往这里灌
- **怎么建**：飞书/钉钉多维表格
- **现在能做吗**：随时可以搭
- **卡在哪**：需要知道企业用什么协作工具
- **要确认的**：企业用飞书还是钉钉？管理者习惯用哪个？
- [ ] **确认：协作工具选择**

## 环节2：错误台账（记录 + 分级 + 追溯责任链）

> 对应需求：F1 决策错误记录、F3 错误分级、F4 多层责任追溯
>
> 入口采集到的错误证据，都汇聚到台账里。台账不只是"记一笔"，还要分级、追溯责任链。

### 做什么

台账的核心字段：

| 字段 | 说明 | 对应需求 |
|------|------|---------|
| 错误描述 | 发生了什么 | F1 |
| 错误来源 | 从哪个入口发现的（说明书核查/库存预警/客诉…） | F1 |
| 严重程度 | 严重/一般（初期两级，后续可细化为三级） | F3 |
| 执行者 | 谁做的 | F4 |
| 审核者 | 谁审的 | F4 |
| 分管领导 | 谁负责的 | F4 |
| 是否重复 | 同一人/同类错误是否之前出现过 | F2 |
| 关联历史 | 如果是重复错误，关联到哪条历史记录 | F2 |
| 发现日期 | 什么时候发现的 | F1 |

### 现在能做吗

技术上随时能搭（飞书多维表格），但有两个规则需要管理者定：

### 需要管理者定的规则

1. **分级标准**：什么算"严重"？什么算"一般"？
   - 建议：初期简单粗暴——造成直接经济损失或客户投诉的算严重，其余算一般
   - [ ] **确认：分级标准**

2. **责任追溯层级**：哪些决策需要追溯审核链？追溯到哪一？
   - 建议：所有录入台账的错误都追溯三层（执行者→审核者→分管领导），填不出来的留空
   - [ ] **确认：追溯规则**

## 环节3：评分卡（自动统计决策质量）

> 对应需求：F2 重复错误识别、F7 年度考核报告
>
> 台账数据自动汇总，生成每个高管的"决策质量评分卡"。

### 做什么

评分卡内容：

| 指标 | 说明 |
|------|------|
| 本季度错误总数 | 台账中该高管关联的错误条数 |
| 严重错误数 | 其中标记为"严重"的条数 |
| 重复错误数 | 其中标记为"重复"的条数（最扎眼的指标） |
| 重复错误明细 | 本次错误 vs 历史错误的对比（让管理者一眼看到"又犯了"） |
| 建议绩效系数 | 根据考核规则自动计算（见环节4） |

### 现在能做吗

台账跑起来之后，评分卡就是一个统计查询，技术上不难。飞书多维表格的仪表盘功能就能做，或者导出数据用Python生成。

### 不需要额外确认

这一步是纯技术活，环节1和环节2跑通了，这里自动就能出。

---

## 环节4：考核挂钩（发动机）

> 对应需求：F5 绩效考核联动
>
> 这是整条链的发动机。没有考核挂钩，前面的记录和评分都是摆设。

### 做什么

评分卡的结果 → 影响绩效/奖金。

管理者原话："最无情的考核就是AI"——去除人情因素，用数据说话。

### 现在能做吗

技术上能做（评分卡数据 → 绩效系数计算），但**规则必须管理者拍板**，技术协助方定不了。

### 需要管理者定的规则

1. **绩效系数怎么算**？
   - 示例：基准1.0，每次严重错误扣0.1，重复错误额外扣0.1
   - [ ] **确认：绩效系数计算方式**

2. **挂钩什么**？年终奖金？工资升降？岗位去留？还是先从奖金开始？
   - 建议：先挂奖金（阻力最小），跑顺后再扩展到岗位评估
   - [ ] **确认：考核挂钩范围**

3. **谁来审核评分结果**？管理者本人？还是HR？
   - 建议：管理者本人终审，确保权威性
   - [ ] **确认：评分审核人**

---

## 环节5：压力传导（终局）

> 对应需求：M3 客观考核、M4 自驱动机制
>
> 这不是一个需要"做"的环节，是环节1-4跑通后的自然结果。

### 为什么会自动实现

- 副总看到自己的评分卡，知道错误会影响绩效 → **副总有动力了**
- 副总的评分卡里有"分管领导"责任 → **副总会主动盯经理**
- 经理知道错误会追溯到审核者 → **经理会主动盯工程师**
- 所有人的评分标准一样 → **没人觉得被针对，制度公平**

### 管理者不用做什么

看评分卡就行。谁的分低，数据说话，不用亲自盯。

| 需求 | 说明 | 为什么现在不做 |
|------|------|---------------|
| M1 集体决策记录 | 重大决策时记录每个参与者的表态 | 需要高管配合录音/记录，现在推会觉得"被监控"。等系统运行半年、大家习惯了被数据评价后再推 |
